# Supabase Database Error Diagnosis

Date: 2026-07-10
Project ref: `iiiyoafpzfqxpaqheojg`

## Conclusion

The repeated `25P02` errors are generated by Supabase's internal
`postgres_exporter`, not by this project's frontend RPC code.

`postgres_exporter` collects PostgreSQL metrics for Supabase monitoring. Its
connection was found in this state:

```text
pid:              6250
user:             supabase_admin
application_name: postgres_exporter
client_addr:      ::1
state:            idle in transaction (aborted)
backend_start:    2026-07-07 10:43:06 UTC
PostgreSQL:       17.6
```

The exporter runs approximately once per minute. After one SQL statement
fails, its transaction remains aborted. Every later statement on that
connection then produces:

```text
25P02 current transaction is aborted, commands ignored until end of transaction block
```

This explains the bursts at every `xx:xx:06` in `supabase_logs.json`.

## Root Cause

PostgreSQL 17 changed the statistics views. In this database,
`pg_stat_bgwriter` contains only:

```text
buffers_clean
maxwritten_clean
buffers_alloc
stats_reset
```

Checkpoint columns were moved to `pg_stat_checkpointer`.

An older exporter query that asks for the pre-PostgreSQL-17 columns fails:

```sql
select checkpoints_timed, checkpoints_req,
       checkpoint_write_time, checkpoint_sync_time,
       buffers_checkpoint, buffers_clean,
       maxwritten_clean, buffers_backend,
       buffers_backend_fsync, buffers_alloc,
       stats_reset
from pg_stat_bgwriter;
```

The database returns:

```text
42703: column "checkpoints_timed" does not exist
```

The same database accepts the later `pg_stat_database_conflicts` query, so
that query is not the original failure. It is a later query running on the
already-aborted exporter transaction.

## What Was Checked

- `supabase_logs.json`: 1000 records, 858 with status `25P02`.
- The errors occur in regular one-minute bursts.
- The active aborted session belongs to `postgres_exporter`.
- No user-created `pg_cron` job was found.
- The project source has no one-minute Supabase polling loop or explicit
  `BEGIN`/`COMMIT` transaction.
- The `pg_stat_database_conflicts` metric query succeeds in a fresh
  transaction.
- The obsolete PostgreSQL 16-era `pg_stat_bgwriter` query reproduces
  `42703` on PostgreSQL 17.6.

## Remediation

The exporter is Supabase-managed. It cannot be upgraded, reconfigured, or
restarted from the project repository or normal SQL Editor permissions.

I attempted to reset only the broken exporter connection with:

```sql
with targets as (
  select pid
  from pg_stat_activity
  where application_name = 'postgres_exporter'
    and state = 'idle in transaction (aborted)'
)
select pid, pg_terminate_backend(pid)
from targets;
```

PostgreSQL rejected it with permission error `42501` because the project
management role cannot terminate the Supabase-owned `supabase_admin`
process.

The permanent fix must be performed by Supabase:

1. Upgrade the internal exporter to a PostgreSQL-17-compatible version.
2. Disable or replace the obsolete `pg_stat_bgwriter` collector.
3. Recycle the exporter connection and ensure failed metric queries are
   rolled back before the next scrape.

A project restart might recycle the connection, but it causes downtime and
does not permanently fix an incompatible exporter. Do not create a fake
compatibility view or alter application RPC code to work around this
internal monitoring problem.
